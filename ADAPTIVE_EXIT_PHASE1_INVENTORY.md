# Adaptive Exit — Phase 1 Inventory: Every Exit-Stack Mechanism And Hardcoded Value

This is the Phase 1 deliverable required by IMPLEMENT_DYNAMIC_ADAPTIVE_EXIT_SYSTEM.md: a complete, line-confirmed inventory of every mechanism in the exit stack and every hardcoded value, with each value classified as either a move-size or geometry threshold that should become volatility-and-fee-derived, or a behavioral gate or true invariant that must stay fixed. It is investigation only; no code, configuration, or flag was changed to produce it. It is written in prose with heading structure for screen-reader access: no tables, no emoji, no decorative separators.

## Method and coverage

The complete exit stack, about fifteen thousand nine hundred lines across fourteen files, was read in full by eight parallel readers, each assigned a file or file group and each confirming current line numbers by reading the file directly rather than trusting any cited line. Every reader recorded that its assigned range was read in full with no sampling. The investigation found 151 mechanisms and 526 hardcoded values in total: 193 classified as move-size or geometry values that should become adaptive, 312 classified as behavioral gates or true invariants that must stay fixed, and 21 marked uncertain and explained. Findings were cross-checked against EXIT_SYSTEMS_DEEP_FORENSIC_FINDINGS.md so nothing it already proved is missed.

Each value below is given with its name, its current location, its current value, its backing configuration key or a note that it is an inline literal, what it controls, what depends on it, the reason it is mistuned for the real move sizes or the reason it is a legitimate invariant, and the proposed adaptive basis. The central deliverable is Part 2, the values that should become adaptive.

## Investigator verification notes

Beyond the parallel readers, the highest-stakes values were re-read directly against the live code and confirmed exactly: the catastrophic per-trade cap (cap_dollar_ceiling 75.0 and cap_pct_of_notional_young 2.5 percent at config.toml lines 2202 and 2203, fee-adjusted by cap_round_trip_fee_pct 0.11 at line 2212); the watchdog hard stop, which is the inline literal "if pnl_pct < -3.0" at src/workers/position_watchdog.py line 2596 (forensic E1, confirmed not config-backed); the ladder arm and rungs (min_profit_to_arm_ladder_pct 0.2 at line 2015, ladder_step_pct_young 0.6 at 1996, ladder_breakeven_lock_pct 0.05 at 2031, ladder_lock_fee_clearance_pct 0.13 at 2047, micro_floor_arm_pct 0.10 at 2060, micro_floor_arm_fee_aware_enabled true at 2078); the trail floor (min_trail_pct 0.30 at 1879, min_trail_atr_multiplier 1.5 at 1878, base_atr_multiplier 2.5 at 1747); the take-profit (default_take_profit_pct 6.0 at 634); and the development-guard tiers in src/risk/layer4_protection.py, where _check_profit returns protected when pnl is above profit_protection_threshold 0.0, protected when pnl is above development_window_lower minus 0.3, and only not-protected when pnl is at or below minus 0.3 percent, which is exactly the dead-drifter trap (forensic F4, G2, C6).

One correction to a reader's framing of the gateway minimum-distance, verified directly in src/core/sl_gateway.py lines 694 to 932 and src/analysis/vol_scale.py. The per-class ceiling min_distance_class_ceiling is dead 0.30, low 0.50, medium 1.00, high 2.00, extreme 3.50, but that ceiling is an UPPER cap on the effective minimum, not a floor. The effective minimum is the maximum of the absolute floor (min_distance_abs_floor_pct, 0.05 percent) and atr_5m_pct times the multiplier (0.5), then capped by the per-class ceiling. So on a dead coin with ATR about 0.04 percent the effective minimum is about 0.05 percent, not 0.30 percent; the dead-class ceiling only binds when ATR exceeds about 0.60 percent, which is not a dead coin. The binding cause of the clamp-noop (forensic B1 and C4) is therefore not the dead ceiling re-imposing 0.30 percent. It is two things confirmed in the code: first, the final tighten-only re-check at lines 898 to 910 rejects the write as a no-op when the clamped value does not improve the current stop; second, the existing breakeven carve-out at lines 743 to 815 holds the stop only down to BREAKEVEN, which does not improve on a stop already at breakeven, so the lock is dropped. The Commit 1 fix is to hold the armed profit lock at its R-derived, fee-floored value (above breakeven) for the trusted profit-lock source, which does improve and therefore writes, while tighten-only, the wrong-side guard, the maximum-step rule, and the rate-limit all remain unchanged.

- Clamp-noop reject (B1) (src/core/sl_gateway.py:898-910). - Terminal wrong-side guard (Issue 2.1) (src/core/sl_gateway.py:912-932).
