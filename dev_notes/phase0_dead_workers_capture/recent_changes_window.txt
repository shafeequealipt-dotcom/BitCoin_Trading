    phase2-post-layer1-fixes: attach LayerManager to underlying OrderService through transformer (audit)
    
    Real-pipeline audit revealed that ``self._services["order_service"]`` is
    a ``_OrderProxy`` (created by ``Transformer.create_proxies()`` for live
    mode switching). The proxy forwards ``place_order`` / ``modify_order``
    / ``cancel_order`` to ``transformer.active_order_service`` but does NOT
    expose ``attach_layer_manager``. The pre-existing manager wiring at
    line 561:
    
        order_svc = self._services.get("order_service") or self._services.get("order")
        if order_svc and hasattr(order_svc, "attach_layer_manager"):
            order_svc.attach_layer_manager(layer_manager)
    
    silently skips because hasattr(_OrderProxy, "attach_layer_manager") is
    False. The underlying bybit OrderService inside transformer._bybit_services
    NEVER gets ``_layer_manager`` set. Pre-fix this was harmless (the legacy
    fail-open allowed orders through). With the Phase 1 (post-Layer-1)
    boot-policy now rejecting entry-side surfaces when ``lm is None``, this
    gap would hard-break live Bybit mode by treating every brain trade as
    a boot-window violation — Layer3BootNotReadyError on the first
    ``layer3_entry``.
    
    Live system today is in shadow mode, where ShadowOrderService has no
    gate (the gate lives only on live OrderService), so this defect was
    latent. But the moment the operator switches to Bybit live, every
    entry-side trade rejects.
    
    Fix: after the proxy attach attempt, walk transformer._bybit_services
    and transformer._shadow_services and call attach_layer_manager on any
    underlying that exposes it. ShadowOrderService doesn't (Shadow has no
    L3 gate by design); BybitOrderService does. Idempotent — repeated
    calls with the same LM are no-ops per attach_layer_manager docstring.

      - WebSocket disconnects can drop position-update events.

        running _state_sync_loop, which reads data/layer_state.json every
        interval and compares to in-memory state.

      - Defensive: corrupt JSON, missing file, phantom keys all handled

    fix(constants): SymbolRegistry now satisfies the full frozenset contract
    
    Root cause: SymbolRegistry was introduced as a dynamic dropin replacement
    for the legacy ``SUPPORTED_SYMBOLS: frozenset[str]`` constant but only
    implemented a fraction of frozenset's surface. The class supported
    ``__contains__`` / ``__iter__`` / ``__len__`` and that was it. Two
    real production / test code paths broke as a result:
    
    1. ``src/strategies/scanner.py:584`` performs
       ``sorted(SUPPORTED_SYMBOLS - TESTNET_EXCLUDED_SYMBOLS)`` to compute
       testnet-eligible symbols. ``__sub__`` was missing → TypeError every
       time ``_scan_testnet`` ran. The branch is reached on testnet config,
       meaning testnet operators got a hard error from the scanner.
    
    2. ``src/trading/services/order_service.py``, ``src/risk/validators.py``,
       ``src/brain/decision_parser.py`` validate every Claude trade against
       ``SUPPORTED_SYMBOLS``. The registry was pre-seeded with only
       {BTCUSDT, ETHUSDT}, so any trade on a known but un-scanned symbol
       (e.g. SOLUSDT, ADAUSDT) was rejected at boot before the first
       scanner tick had a chance to populate the registry. Order qty
       tables (MIN_ORDER_QTY / MAX_ORDER_QTY) had entries for these
       symbols, the registry didn't.
    
    Fix (industry-standard, root cause):
    
    * SymbolRegistry now implements the full set-algebra contract via
      ``snapshot()``-based dunders that return frozenset point-in-time
      copies: ``__sub__``, ``__rsub__``, ``__or__``, ``__ror__``,
      ``__and__``, ``__rand__``, ``__xor__``, ``__rxor__``, ``__le__``,
      ``__ge__``, ``__eq__``. Snapshot semantics match what frozenset
      callers depended on — the result survives subsequent ``update()``
      calls. ``__hash__`` raises explicitly so the registry can't be
      used as a dict key by mistake (callers must ``.snapshot()`` first).
    
    * New ``snapshot()`` / ``add()`` methods. ``_coerce`` helper accepts
      any iterable for set-op rhs (mirrors frozenset behavior).
    
    * Boot-time invariant restored: every key in MIN_ORDER_QTY /
      MAX_ORDER_QTY is added to SUPPORTED_SYMBOLS at module-load time so
      order/risk validation works for known symbols before the first
      scanner cycle. Loop runs once at import; the registry is mutable
      but seeded deterministically from the order-qty tables.
    
    * Updated ``test_is_frozenset`` → ``test_supports_legacy_frozenset_contract``
      asserts the CONTRACT (membership, iteration, conversion, set-algebra
      results, equality), not the type. Type-asserting against a class we
      intentionally replaced was the test bug.
